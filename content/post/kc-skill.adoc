For the last ten years I've stumbled my way through the JVM-centric world of "big data" (as it was called then), relying on my wits with SQL and config files to just about muddle through.
One of the things that drew me to Kafka Connect was that I could build integrations between Kafka and other systems without needing to write Java, and the same again for ksqlDB and Flink SQL—now stream processing was available to mere mortals and not just the Java adonis'.

One thing defeated me though; if a connector _didn't exist_ for Kafka Connect, then I was stuck.
I'd resort to cobbled-together pipelines leaning heavily on +++<del>kafkacat</del>+++ kcat, such as I did in https://www.confluent.io/blog/streaming-etl-and-analytics-for-real-time-location-tracking/[this blog post].
I built some cool analytics on top of maritime AIS data about ships' locations, but the foundations were shaky at best:

[source,bash]
----
nc ais.host.com 5631 | gpsdecode | kafkacat -b broker:9092 -P
----

No failure logic, no schema handling, no bueno.

What I really needed was a connector for Kafka Connect.
However for that, you need Java.
I don't write Java.

But Claude can write Java.

Great questions. Let me address both.

**TCP pipe architecture**: Yes, this changes things meaningfully. Most source connectors follow a request/response pattern (query DB → get rows → return records). A live TCP stream is fundamentally different:

- **No replay**: If the connector is down, messages are lost. The offset is for observability only, not resume.
- **Long-lived connection**: The TCP socket persists across `poll()` calls (opened in `start()`, read in `poll()`, closed in `stop()`). This is similar to MQTT or syslog source connectors.
- **No backoff needed**: Data flows continuously — `poll()` just reads whatever's buffered on the socket. No "sleep when no data" pattern (though `SO_TIMEOUT` handles the case where the stream pauses).
- **No snapshot phase**: No existing data to catch up on.
- **Reconnection is critical**: TCP connections drop. This is the main failure mode to handle well.

The architecture is sound — it's just a different *flavor* of source connector. The Connect framework handles this fine.

**Topic strategy**: You're right, 1:N is worth considering. With separate topics, each gets a tighter schema (no nullable fields from other types). Let me think through the options:

- **Single topic** (`ais`): simpler, matches your existing ksqlDB pipeline
- **Split by category** (`ais.position`, `ais.static`, `ais.base_station`, `ais.other`): cleaner schemas, easier for consumers that only care about one type
- **Configurable**: best of both worlds
****

Good questions. Let me address both.

**Message headers**: Yes, headers are useful here. The key win is `ais.msg_type` — consumers can filter/route by header without deserializing the value. Also useful for provenance (source station, raw NMEA for debugging).

**TCP reconnection/stall prevention**: The critical insight is that `poll()` must never block indefinitely. Here's the layered defense:

1. **`SO_TIMEOUT`** on the socket (e.g., 1000ms) — `readLine()` throws `SocketTimeoutException` after 1s of no data, so `poll()` always returns
2. **`stop()` closes the socket** from another thread — immediately unblocks any pending `readLine()` with an IOException
3. **Volatile `stopping` flag** — `poll()` checks it on each iteration
4. **Backoff on reconnect** — avoids hammering a down endpoint

Let me update the plan with both.

== The Connector

image:/images/2026/03/kafka-connect-ais-q80.webp[]

(Credit: Claude, obvs)

Let's start up Kafka, Kafka Connect, and Schema Registry:

[source,bash]
----
❯ docker compose up -d --build
[+] Running 4/4
 ✔ Network kafka-connect-ais_default  Created   0.0s
 ✔ Container broker                   Started   0.4s
 ✔ Container schema-registry          Started   0.4s
 ✔ Container kafka-connect            Started   0.4s
----

Configure https://github.com/kcctl/kcctl[kcctl] for easy working with Kafka Connect:

[source,bash]
----
kcctl config set-context --cluster=http://localhost:8083 local
----

Check that the connector plugin has loaded (my connector! that I 'wrote'! omg!!):

[source,bash]
----
❯ kcctl get plugins | grep -i ais
 source             net.rmoff.connect.ais.AisSourceConnector                           0.1.0
----

The bare config is very simple:

[source,javascript]
----
{
  "name": "ais-source-per-type",
  "config": {
    "connector.class": "net.rmoff.connect.ais.AisSourceConnector",
    "ais.hosts": "153.44.253.27:5631",
    "topic": "ais",
    "topic.per.type": "true",
    "tasks.max": "1"
  }
}
----

Create an instance of the connector using this config:

[source,bash]
----
kcctl apply -f connector-ais-per-type.json
----

Check the connector is working:

[source,bash]
----
❯ kcctl get connectors

 NAME                  TYPE     STATE     TASKS
 ais-source-per-type   source   RUNNING   0: RUNNING <1>
----
